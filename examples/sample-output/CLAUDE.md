## Architecture

Hexagonal on the backend (domain / application / infrastructure). iOS client
uses MVVM. Communication over REST + a signed webhook channel for fulfillment
callbacks. PostgreSQL for persistence with SQLAlchemy 2.x (async). No ORM in
the domain layer — repositories live at the infrastructure boundary.

### Modules

- `backend/app/domain/` — pure business rules, no I/O
- `backend/app/application/` — use cases that orchestrate domain + repositories
- `backend/app/infrastructure/` — DB, HTTP, external APIs
- `backend/app/api/` — FastAPI routers, thin glue
- `ios-app/OrdergateApp/` — SwiftUI views + view models
- `ios-app/OrdergateCore/` — shared models and API client

## Rules

### Claude must ALWAYS
- Use type hints in Python (enforced by `ruff` and `mypy --strict`).
- Run `pytest -q` before suggesting a commit.
- Keep domain layer free of framework imports (no `fastapi`, no `sqlalchemy`).
- Use conventional commit messages.
- Add a migration file for any schema change (`alembic revision --autogenerate`).

### Claude must NEVER
- Commit to `main` directly — always PR.
- Modify `backend/app/domain/` without a matching test.
- Touch `ops/secrets/` or any file under `infra/terraform/state/`.
- Run `alembic downgrade` without explicit user confirmation.

### Known pitfalls
- The webhook signature verification is timing-sensitive; use
  `hmac.compare_digest`, never `==`.
- `OrderID` collisions have happened twice — always use the
  `generate_order_id()` helper, never `uuid4()` directly.
- The test DB is truncated between tests via a session-scoped fixture; adding
  a new test module requires importing `conftest` from the parent.
